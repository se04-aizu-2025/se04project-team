# ドキュメント作成ベストプラクティス

良いドキュメントは「読者が何に困っているか」を起点に書く。

---

## 核心原則: 読者視点で考える

ドキュメントを書く前に、以下を自問する：

| 質問 | 悪い例 | 良い例 |
|------|--------|--------|
| **誰が読むのか？** | 誰でも | 新規参入の開発者 |
| **何がわからないのか？** | 全体 | このモジュールの追加方法 |
| **読んだ後どうなりたいのか？** | 理解したい | 自分で画面を追加できる |

> [!IMPORTANT]
> 読者は「理解したい」のではなく「できるようになりたい」のである。

---

## アンチパターン: 避けるべき書き方

### ❌ 自明なことを書く

```markdown
<!-- 悪い例 -->
## UserRepository
UserRepositoryはUserに関するRepositoryです。

<!-- 良い例 -->
## UserRepository
ユーザーデータの永続化を担当。キャッシュ戦略についてはXXXを参照。
```

**判定基準**: コードを読めばわかることは書かない。

### ❌ 目的なく網羅する

```markdown
<!-- 悪い例 -->
### プロパティ一覧
| 名前 | 型 | 説明 |
| id | Int | ID |
| name | String | 名前 |
| createdAt | Date | 作成日時 |

<!-- 良い例 -->
### 注意が必要なプロパティ
- `status`: 状態遷移図は [こちら](./状態遷移.md)
- `deletedAt`: 論理削除。nullでないレコードはクエリから除外される
```

**判定基準**: 読者が迷いそうな箇所だけ書く。

### ❌ 抽象的な説明に終始する

```markdown
<!-- 悪い例 -->
MVIパターンを採用しています。Stateは不変で、Intentで状態遷移します。

<!-- 良い例 -->
MVIパターンを採用。以下のファイルを作成する：
1. `XxxState.kt` - 画面状態を定義
2. `XxxIntent.kt` - ユーザー操作を定義
3. `XxxViewModel.kt` - IntentをStateに変換

具体例: `SortScreen` の実装を参照
```

**判定基準**: 読者が「で、どうすればいい？」と思わないか。

---

## パターン: 効果的な書き方

### ✅ 問い→答え の構造

読者が持つであろう疑問を見出しにする：

```markdown
## 新しい画面はどこに追加する？
`presentation/feature/` 配下に新規モジュールを作成。

## 既存のコンポーネントを使いたい
`designsystem/components/` を確認。一覧は [こちら](./DESIGN_SYSTEM.md)。

## テストはどう書く？
命名規則: `given_状況_when_操作_then_結果`
```

### ✅ すぐ使える例を載せる

```markdown
## 使用例

```kotlin
// ✅ Good
SortButton(onClick = { viewModel.onIntent(StartIntent) }, text = "開始")

// ❌ Bad - Material3を直接使用
Button(onClick = { }) { Text("開始") }
```

### ✅ 判断基準を明確にする

```markdown
## いつ SortCard を使うか？
- クリック可能なカード → `SortCard(onClick = ...)`
- セクション区切り → `SortSectionCard(title = ...)`
- 情報表示のみ → `SortSurface`
```

### ✅ 関連リンクで深堀りを促す

```markdown
## 関連ドキュメント
- アーキテクチャ全体: [ARCHITECTURE.md](./ARCHITECTURE.md)
- デザイントークン: [DESIGN_SYSTEM.md](./DESIGN_SYSTEM.md)
```

---

## チェックリスト

ドキュメントを書いたら確認：

- [ ] **読者を定義したか？** 誰向けか明確
- [ ] **ゴールを定義したか？** 読後に何ができるようになるか明確
- [ ] **自明な記述はないか？** コードを読めばわかることを書いていない
- [ ] **具体例はあるか？** Good/Badの例、またはコードスニペット
- [ ] **次のアクションは明確か？** 読者が次に何をすればいいかわかる
- [ ] **リンクは貼ったか？** 詳細は別ドキュメントへ誘導

## ドキュメントの種類と目的

| 種類 | 目的 | 読者 | 含めるべき内容 |
|------|------|------|---------------|
| **AGENTS.md** | 全体把握 | AI・新規開発者 | 技術スタック、実行コマンド、現状 |
| **ONBOARDING.md** | 入門 | 新規開発者 | 環境構築、最初のPR作成まで |
| **ARCHITECTURE.md** | 設計理解 | 既存開発者 | レイヤー構成、依存ルール、判断基準 |
| **DESIGN_SYSTEM.md** | UI実装 | フロント開発者 | トークン、コンポーネント一覧、使い分け |
| **FUNDAMENTALS.md** | 規約確認 | 全開発者 | 命名規則、禁止事項、チェックリスト |

---

## まとめ

**良いドキュメントとは:**

1. 読者の「困っていること」を解決する
2. 「で、どうすればいい？」に答えている
3. 読んだ後に行動できる

**書く前に問え:**

> この内容がなかったら、読者は何に困るのか？

# プルリクエスト (PR) ガイド

このドキュメントでは、プルリクエストの作成方法とレビュープロセスについて説明します。

## PRテンプレート

プルリクエスト作成時、[`.github/pull_request_template.md`](../.github/pull_request_template.md) が自動的に適用されます。

---

## PRの書き方

### 1. 概要

PRの目的を簡潔に説明してください。何を解決しようとしているのか、なぜこの変更が必要なのかを明確にします。

### 2. 変更内容

主要な変更点をリストアップしてください：

```markdown
- 機能Aを追加
- バグBを修正
- ファイルCをリファクタリング
```

### 3. 変更の種類

該当するものにチェックを入れます：

| 絵文字 | 種類 | 説明 |
|--------|------|------|
| 🐛 | Bug fix | バグ修正 |
| ✨ | New feature | 新機能 |
| 💄 | UI/UX | UIやスタイルの変更 |
| ♻️ | Refactoring | リファクタリング |
| 📝 | Documentation | ドキュメント更新 |
| 🧪 | Test | テストの追加・修正 |
| 🔧 | Configuration | 設定ファイルの変更 |
| ⚡️ | Performance | パフォーマンス改善 |

### 4. 関連Issue

関連するIssueがあれば記載します：

```markdown
Closes #123      <!-- PRマージ時にIssueを自動クローズ -->
Relates to #456  <!-- 関連するが自動クローズしない -->
```

### 5. テスト方法

この変更をどのようにテストしたか、再現手順を記載してください。

### 6. スクリーンショット/動画

UIの変更がある場合は、変更前後のスクリーンショットまたは動画を添付してください。

---

## PRチェックリスト

PRを提出する前に、以下を確認してください：

- [ ] コードが本プロジェクトのコーディング規約に従っている
- [ ] 自分のコードをセルフレビューした
- [ ] コードに必要なコメントを追加した
- [ ] ドキュメントを更新した（必要な場合）
- [ ] 新しい警告が発生していない
- [ ] テストを追加した（必要な場合）
- [ ] すべてのテストがパスした
- [ ] ビルドが成功している

---

## CI/CD チェック

PRを作成すると、以下の自動チェックが実行されます：

- **ktlint** - コードスタイルチェック
- **detekt** - 静的解析
- **テスト** - ユニットテストの実行
- **ビルド** - 各プラットフォームのビルド

> [!IMPORTANT]
> すべてのCIチェックがパスするまで、PRはマージできません。

---

## レビュープロセス

1. **PRを作成** - テンプレートに従って記入
2. **CIチェック** - 自動チェックの完了を待つ
3. **レビュー依頼** - チームメンバーにレビューを依頼
4. **フィードバック対応** - レビューコメントに対応
5. **承認 & マージ** - 承認後にマージ
